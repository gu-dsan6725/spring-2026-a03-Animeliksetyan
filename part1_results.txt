# Authentication Flow: Token Validation to User Authorization

Based on the provided context, here's how the authentication flow works:

## Token Validation

**Bearer Token Validation** (`registry/auth/internal.py`, lines 105-107):
- The system validates Bearer JWT tokens signed with `SECRET_KEY`
- The validation function `_validate_bearer_token(auth_header: str)` extracts and validates the token from the Authorization header (line 105-106)
- Tokens include a `"token_use": "access"` claim indicating their purpose (`registry/auth/internal.py`, line 60)

## Token Generation

**Internal Token Generation** (`registry/auth/internal.py`, lines 31-60):
- The `generate_internal_token()` function creates JWT tokens internally
- Tokens are signed with the application's secret key

**Rate Limiting** (`auth_server/server.py`, lines 70-71):
- Token generation is rate-limited using an in-memory counter (`user_token_generation_counts`)
- This prevents abuse of token generation endpoints

## Authorization Flow

**Static Token Authentication** (`auth_server/server.py`, lines 77-78):
- The system supports static API key authentication as an alternative to IdP JWT for Registry API access
- This is configurable via `_registry_static_token_requested`

**Middleware Protection** (`registry/middleware/mode_filter.py`, line 30):
- Token endpoints (`/api/tokens/`) are protected by middleware filtering
- This ensures only authorized requests reach the token generation logic
